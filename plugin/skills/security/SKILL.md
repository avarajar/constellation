# Constellation — Security Generation

Generate security hardening configuration for the project.

## What You Need

Either read from the Constellation blueprint, or accept these inputs:
- **Backend framework** and **language** (to apply framework-specific security)
- **Frontend framework** (for CSP and XSS prevention)
- **Cloud provider** (for cloud-specific security configs)

## STEP 0 — MANDATORY: Fetch Latest Versions BEFORE Writing Any Code

**DO NOT SKIP THIS STEP.**

For Node.js:
```bash
for pkg in helmet cors express-rate-limit csurf hpp; do
  ver=$(npm view "$pkg" version 2>/dev/null)
  [ -n "$ver" ] && echo "$pkg@$ver"
done
```

For Python:
```bash
for pkg in django-csp django-ratelimit python-jose cryptography; do
  ver=$(curl -s "https://pypi.org/pypi/$pkg/json" 2>/dev/null | python3 -c "import sys,json; print(json.load(sys.stdin)['info']['version'])" 2>/dev/null)
  [ -n "$ver" ] && echo "$pkg==$ver"
done
```

## What You Generate

### HTTP Security Headers
- **Node.js**: `helmet` middleware with configured options:
  - Content-Security-Policy
  - X-Content-Type-Options: nosniff
  - X-Frame-Options: DENY
  - Strict-Transport-Security
  - X-XSS-Protection
  - Referrer-Policy
- **Python (Django)**: SecurityMiddleware + django-csp settings:
  - `SECURE_BROWSER_XSS_FILTER = True`
  - `SECURE_CONTENT_TYPE_NOSNIFF = True`
  - `X_FRAME_OPTIONS = "DENY"`
  - `SECURE_HSTS_SECONDS` for production
  - CSP directives
- **Python (FastAPI/Flask)**: Security headers middleware

### Rate Limiting
- **Node.js**: `express-rate-limit` or `@nestjs/throttler`:
  - Global: 100 requests per 15 minutes per IP
  - Auth endpoints: 5 requests per 15 minutes (brute force protection)
  - API endpoints: 60 requests per minute
- **Python**: `django-ratelimit` or `slowapi`:
  - Same tier structure
  - Redis-backed for distributed rate limiting

### Input Validation & Sanitization
- Request body validation on all endpoints (already in CRUD, but ensure strict schemas)
- SQL injection prevention (parameterized queries — ORMs handle this, but verify raw queries)
- XSS prevention:
  - **Backend**: sanitize HTML in user inputs before storage
  - **Frontend**: escape all rendered user content (React/Vue do this by default, but flag `dangerouslySetInnerHTML` / `v-html`)
- Path traversal prevention in file operations
- Mass assignment protection (whitelist allowed fields)

### CORS Configuration
- Strict CORS setup (not `*` in production):
  - Whitelist specific origins
  - Limit allowed methods and headers
  - Environment-specific origins (dev: localhost, staging: staging URL, prod: production URL)

### Authentication Security (if auth structure exists)
- Password hashing with bcrypt (cost factor 12+)
- JWT best practices:
  - Short expiration (15 min access, 7 day refresh)
  - HttpOnly, Secure, SameSite cookies for tokens
  - Token rotation on refresh
- Session security (if session-based):
  - Secure cookie flags
  - Session timeout
  - CSRF protection

### Dependency Security
- `npm audit` / `pip-audit` in CI pipeline
- `.npmrc` with `audit=true`
- `safety` or `pip-audit` for Python
- Dependabot/Renovate for automated security updates (generated by common skill)

### Secrets Management
- `.env` file with strict `.gitignore` entry
- No secrets in code — all from environment
- `.env.example` with placeholder values (no real secrets)
- Git hooks to prevent committing secrets:
  - `detect-secrets` in pre-commit
  - `gitleaks` in CI

### Security Checklist in README
Add a security section to README:
- [ ] Environment variables configured (no hardcoded secrets)
- [ ] CORS configured for production origins
- [ ] Rate limiting enabled
- [ ] Security headers configured
- [ ] Dependencies audited (`npm audit` / `pip-audit`)
- [ ] Pre-commit hooks installed (detect-secrets)

## Guidelines

- Security configs should have sensible defaults that work out of the box
- Development mode should be permissive (localhost CORS, relaxed rate limits)
- Production mode should be strict
- Never generate real secrets — always use placeholders and environment variables
- Security should not break the developer experience — things must still work locally

## Standalone Usage

To add security hardening to an existing project:

1. Detect the backend framework and language
2. **Run Step 0 to fetch latest versions**
3. Add security middleware, headers, rate limiting, and CORS config
4. Add security audit to CI pipeline
5. Update README with security checklist
